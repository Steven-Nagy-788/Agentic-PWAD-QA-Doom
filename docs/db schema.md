# Doom Agentic Testing System — Complete Database Schema

## Preliminary Notes Before the Schema

### On tetsuo-doom MCP repo changes needed for PWADs

Looking at the real `start_game` tool signature from the README and the ViZDoom API docs, here is the truth:

**What tetsuo-doom currently does:**
The `start_game` tool accepts a `wad` parameter that takes either a preset string (`"freedoom1"`, `"freedoom2"`) or an **absolute path** to a WAD file. Internally it calls ViZDoom's two separate methods:
- `game.set_doom_game_path(path)` → the **IWAD** (base game assets: textures, sounds, sprites)
- `game.set_doom_scenario_path(path)` → the **PWAD** (the custom map geometry)

**The problem for your use-case:**
The existing `wad` parameter in tetsuo-doom maps to `set_doom_scenario_path` only (the scenario/PWAD slot). If a user uploads a pure PWAD (a custom map that depends on doom2/freedoom2 textures), you need:
1. A base IWAD still set via `set_doom_game_path` (freedoom2.wad by default, already in ViZDoom install)
2. The uploaded PWAD set via `set_doom_scenario_path` with its saved absolute path on disk

**What you must add to your fork of tetsuo-doom:**

In `game_manager.py`, modify `start_game` to accept two parameters:
- `iwad_path: str = None` — path to base IWAD (defaults to freedoom2.wad if None)
- `pwad_path: str` — absolute path to the uploaded PWAD

And in `server.py`, expose both in the `start_game` MCP tool. This is a small, surgical change — about 10-15 lines. The rest of the repo (state.py, actions.py, objects.py) does not need changes.

### On Celery

**You do not need Celery.** Here is why:

Celery is for distributing work across multiple workers asynchronously, typically when you have many concurrent jobs. Your system will run **one game session at a time** (ViZDoom is a heavy process — one instance per machine is realistic for a graduation project). The entire game loop runs in a single Python async background task (FastAPI's `BackgroundTasks` or an `asyncio.create_task`). The agent loop, frame capture, and data collection all happen in that one long-running coroutine. Celery would add Redis as a broker dependency for zero real benefit here. Skip it entirely.

### On Vector DB / Memory

**You do not need a vector database.** Here is the honest breakdown:

- The agent (Gemini 2.5 Flash) gets its context from each MCP tool call response — that is its "memory" within a run. The game state is fully structured JSON per tick.
- You are NOT doing semantic search across past runs. Users view past runs by run ID from the list — that is a simple `SELECT` by foreign key.
- The LLM report generation gets the full structured data dump for one run in a single prompt — no retrieval needed.
- A vector DB would only make sense if you were letting users ask natural language questions like "find me runs where the agent died in the red room" — that is not in your feature set.

**Plain PostgreSQL handles everything.**

## The Complete Schema

### Design Decisions

- All timestamps are `TIMESTAMPTZ` (timezone-aware). Your server is in Cairo, EG (UTC+2/3).
- All IDs are `UUID` generated by PostgreSQL (`gen_random_uuid()`). Never sequential integers for external-facing entities.
- JSONB columns are used only where the structure is genuinely variable per row (game events, static analysis results). Fixed known fields always get their own columns.
- No soft deletes. If a run is deleted, cascade everything.
- `game_events` will be the largest table — one row per agent tick. Expect 500–5000 rows per run depending on how long the agent plays. This is fine for PostgreSQL — no partitioning needed at this scale.

### Table 1: `wad_files`

Stores metadata about every WAD file uploaded. The binary is stored on disk; this table stores the path.

```sql
CREATE TABLE wad_files (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    original_filename   VARCHAR(255)    NOT NULL,
    stored_path         TEXT            NOT NULL UNIQUE,  -- absolute path on disk, e.g. /data/wads/{uuid}.wad
    file_size_bytes     BIGINT          NOT NULL,
    sha256_hash         CHAR(64)        NOT NULL UNIQUE,  -- prevents duplicate uploads of identical files
    uploaded_at         TIMESTAMPTZ     NOT NULL DEFAULT NOW(),

    -- WAD format validation results (set after upload, before analysis)
    -- 'pending' | 'valid_pwad' | 'valid_iwad' | 'invalid' | 'udmf_unsupported'
    validation_status   VARCHAR(32)     NOT NULL DEFAULT 'pending',
    validation_error    TEXT,           -- NULL if valid, error message if not

    -- Detected map names from omgifol (e.g. ARRAY['MAP01','MAP02'] or ARRAY['E1M1'])
    -- Populated during static analysis phase, NULL until then
    detected_maps       TEXT[]
);
```

**Why `sha256_hash`?** If the same WAD is uploaded twice, you can return the existing record instead of storing a duplicate 20MB file. This is a real UX improvement.

**Why `detected_maps TEXT[]`?** omgifol can read all map lumps from a WAD. A PWAD might contain multiple maps. You store this here so the frontend can let the user pick which map to test before starting a run.

### Table 2: `static_analysis_results`

One row per (wad_file, map_name) combination. A WAD with 3 maps produces 3 rows.

```sql
CREATE TABLE static_analysis_results (
    id                      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    wad_file_id             UUID            NOT NULL REFERENCES wad_files(id) ON DELETE CASCADE,
    map_name                VARCHAR(16)     NOT NULL,   -- e.g. 'MAP01', 'E1M1'

    -- Populated by omgifol
    -- Thing counts (enemies, items, keys, weapons, decorations)
    thing_count_total       INTEGER         NOT NULL DEFAULT 0,
    thing_count_enemies     INTEGER         NOT NULL DEFAULT 0,
    thing_count_items       INTEGER         NOT NULL DEFAULT 0,  -- health + armor + powerups
    thing_count_keys        INTEGER         NOT NULL DEFAULT 0,
    thing_count_weapons     INTEGER         NOT NULL DEFAULT 0,
    -- Map geometry
    linedef_count           INTEGER         NOT NULL DEFAULT 0,
    sector_count            INTEGER         NOT NULL DEFAULT 0,
    secret_sector_count     INTEGER         NOT NULL DEFAULT 0,
    vertex_count            INTEGER         NOT NULL DEFAULT 0,
    -- Derived: bounding box of all vertices
    map_width_units         INTEGER,        -- max_x - min_x in Doom map units
    map_height_units        INTEGER,        -- max_y - min_y in Doom map units

    -- Populated by dmon
    total_monster_hp        INTEGER,        -- sum of all monster HP values
    total_health_pickup_pts INTEGER,        -- health pickups converted to HP points
    total_armor_pickup_pts  INTEGER,        -- armor pickups converted to points
    hitscanner_percent      NUMERIC(5,2),   -- % of enemies that are hitscanners (zombies, chaingunners)
    health_ratio            NUMERIC(8,4),   -- health_pts / total_monster_damage (dmon's balance metric)
    ammo_ratio              NUMERIC(8,4),   -- ammo count vs monster HP ratio from dmon
    -- dmon's difficulty label: 'easy' | 'fair' | 'hard' | 'slaughter' | 'unknown'
    estimated_difficulty    VARCHAR(16),

    -- Full raw breakdown as JSONB for detailed frontend display
    -- Keyed by enemy type name: {"IMP": {"count": 5, "hp": 60, "total_hp": 300}, ...}
    enemy_breakdown         JSONB           NOT NULL DEFAULT '{}',
    -- Keyed by item type: {"STIM": {"count": 3, "value": 10}, "MEDI": {...}, ...}
    item_breakdown          JSONB           NOT NULL DEFAULT '{}',

    -- Map overview PNG — absolute path on disk
    map_overview_png_path   TEXT,           -- NULL until rendered

    analyzed_at             TIMESTAMPTZ     NOT NULL DEFAULT NOW(),

    UNIQUE (wad_file_id, map_name)
);
```

**Why separate from `wad_files`?** A WAD can have multiple maps. Analysis runs per map, not per file. The UNIQUE constraint on `(wad_file_id, map_name)` enforces this correctly.

### Table 3: `test_runs`

The central table. One row per agent test session.

```sql
CREATE TABLE test_runs (
    id                      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    wad_file_id             UUID            NOT NULL REFERENCES wad_files(id) ON DELETE RESTRICT,
    static_analysis_id      UUID            REFERENCES static_analysis_results(id) ON DELETE SET NULL,
    map_name                VARCHAR(16)     NOT NULL,

    -- Configuration chosen at run start
    difficulty_level        SMALLINT        NOT NULL DEFAULT 3  CHECK (difficulty_level BETWEEN 1 AND 5),
    -- IWAD used as base (always 'freedoom2' for this project — recorded for reproducibility)
    iwad_used               VARCHAR(64)     NOT NULL DEFAULT 'freedoom2',
    -- LLM model identifier — recorded so past runs show what model was used
    llm_model               VARCHAR(128)    NOT NULL DEFAULT 'gemini-2.5-flash',

    -- Run lifecycle
    -- 'pending' | 'analyzing' | 'running' | 'completed' | 'failed' | 'cancelled'
    status                  VARCHAR(16)     NOT NULL DEFAULT 'pending',
    started_at              TIMESTAMPTZ,    -- set when ViZDoom init completes
    completed_at            TIMESTAMPTZ,    -- set when game ends (death, map exit, timeout, or error)
    duration_seconds        INTEGER,        -- computed at end: EXTRACT(EPOCH FROM completed_at - started_at)

    -- Outcome (set on completion)
    -- 'map_completed' | 'player_died' | 'timeout' | 'error' | 'cancelled'
    outcome                 VARCHAR(32),
    error_message           TEXT,           -- populated if status = 'failed'

    -- High-level gameplay stats (aggregated from game_events at run end)
    final_hp                SMALLINT,       -- player HP at run end (0 if died)
    final_armor             SMALLINT,
    total_kills             SMALLINT,
    total_deaths            SMALLINT,       -- number of times player died (with respawn)
    secrets_found           SMALLINT,
    total_items_collected   SMALLINT,
    total_actions_taken     INTEGER,        -- number of MCP step() calls made by agent
    total_llm_calls         INTEGER,        -- number of times Gemini API was called

    -- File paths (all absolute paths on disk)
    recording_mp4_path      TEXT,           -- NULL until recording is finished and muxed
    report_pdf_path         TEXT,           -- NULL until report generation completes

    created_at              TIMESTAMPTZ     NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_test_runs_wad_file_id ON test_runs(wad_file_id);
CREATE INDEX idx_test_runs_status ON test_runs(status);
CREATE INDEX idx_test_runs_created_at ON test_runs(created_at DESC);
```

**Why `ON DELETE RESTRICT` on wad_file_id?** You cannot delete a WAD file if runs reference it. This protects video recordings and reports from becoming orphaned.

**Why record `llm_model`?** If you upgrade from gemini-2.5-flash to something else mid-project, past runs still show exactly which model played them. Critical for academic reproducibility.

### Table 4: `game_events`

The raw tick-by-tick data stream. This is the most important table for analytics and the report.

```sql
CREATE TABLE game_events (
    id                  BIGSERIAL       PRIMARY KEY,   -- BIGSERIAL not UUID: high insert rate, no external exposure
    run_id              UUID            NOT NULL REFERENCES test_runs(id) ON DELETE CASCADE,
    tick_number         INTEGER         NOT NULL,       -- ViZDoom episode tic counter (from get_episode_time())
    recorded_at         TIMESTAMPTZ     NOT NULL DEFAULT NOW(),

    -- Player state (from ViZDoom GameVariables)
    player_x            REAL            NOT NULL,
    player_y            REAL            NOT NULL,
    player_angle        SMALLINT        NOT NULL,       -- degrees 0-359
    health              SMALLINT        NOT NULL,
    armor               SMALLINT        NOT NULL,
    ammo_bullets        SMALLINT        NOT NULL,
    ammo_shells         SMALLINT        NOT NULL,
    ammo_rockets        SMALLINT        NOT NULL,
    ammo_cells          SMALLINT        NOT NULL,
    kill_count          SMALLINT        NOT NULL,
    item_count          SMALLINT        NOT NULL,
    secret_count        SMALLINT        NOT NULL,
    weapon_selected     SMALLINT        NOT NULL,       -- ViZDoom WEAPON game variable

    -- Agent decision (what the LLM decided to do this tick)
    -- NULL if this tick was not an LLM-decision tick (throttled ticks)
    action_taken        JSONB,                          -- e.g. {"MOVE_FORWARD_BACKWARD_DELTA": 10, "ATTACK": 1}
    llm_reasoning       TEXT,                           -- raw chain-of-thought from Gemini (can be long)

    -- Event classification (derived by your data collector, not from ViZDoom directly)
    -- 'normal' | 'kill' | 'death' | 'item_pickup' | 'secret_found' | 'damage_taken' | 'map_exit'
    event_type          VARCHAR(32)     NOT NULL DEFAULT 'normal',

    -- Populated only for kill events: which enemy type was killed
    killed_enemy_type   VARCHAR(64),
    -- Populated only for damage_taken events
    damage_received     SMALLINT,

    UNIQUE (run_id, tick_number)
);

CREATE INDEX idx_game_events_run_id ON game_events(run_id);
CREATE INDEX idx_game_events_run_id_tick ON game_events(run_id, tick_number);
-- Partial index for event analysis — only indexes meaningful events, not every normal tick
CREATE INDEX idx_game_events_notable ON game_events(run_id, event_type)
    WHERE event_type != 'normal';
```

**Why BIGSERIAL not UUID?** This table will have thousands of inserts per run, every run. BIGSERIAL is an 8-byte integer — far cheaper for sequential inserts and index maintenance than a 16-byte random UUID. The `id` here is never exposed externally; it is only for internal DB mechanics.

**Why store `llm_reasoning` here?** This is gold for your report. You can quote specific Gemini decisions in the report — "At tick 342, the agent observed 3 enemies and chose to retreat: [reasoning]". This also lets you analyze whether the agent's reasoning quality correlates with performance.

**Why not store the screenshot PNG per tick?** At 30 ticks/second over a 5-minute run that is 9000 PNG files. Instead, you capture video via OpenCV `VideoWriter` from the framebuffer. Screenshots are only stored at notable events (deaths, kills) if you want them.

### Table 5: `notable_event_screenshots`

Screenshots captured only at meaningful moments. Not every tick.

```sql
CREATE TABLE notable_event_screenshots (
    id              UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    run_id          UUID        NOT NULL REFERENCES test_runs(id) ON DELETE CASCADE,
    game_event_id   BIGINT      NOT NULL REFERENCES game_events(id) ON DELETE CASCADE,
    screenshot_path TEXT        NOT NULL,   -- absolute path to PNG on disk
    captured_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_screenshots_run_id ON notable_event_screenshots(run_id);
```

### Table 6: `agent_position_trail`

A downsampled position log for rendering the heatmap/path overlay on the map PNG. You do NOT store every tick here (that is in `game_events`). You store every Nth tick (e.g. every 10 ticks) purely for map visualization.

```sql
CREATE TABLE agent_position_trail (
    id          BIGSERIAL   PRIMARY KEY,
    run_id      UUID        NOT NULL REFERENCES test_runs(id) ON DELETE CASCADE,
    tick_number INTEGER     NOT NULL,
    x           REAL        NOT NULL,
    y           REAL        NOT NULL,
    -- Snapshot of HP at this position sample (used for color-coding path: red=low hp, green=high hp)
    health      SMALLINT    NOT NULL
);

CREATE INDEX idx_position_trail_run_id ON agent_position_trail(run_id);
```

**Why a separate table and not just query `game_events`?** The map overlay is rendered client-side from this data. If you query 5000 game_events rows and send them to the frontend just for a path overlay, that is wasteful. This table stores only the ~500 downsampled points needed for the visual.

### Table 7: `test_reports`

The generated QA report, structured to match your template sections.

```sql
CREATE TABLE test_reports (
    id                          UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    run_id                      UUID        NOT NULL UNIQUE REFERENCES test_runs(id) ON DELETE CASCADE,

    -- Section 1: Introduction
    report_purpose              TEXT,
    intended_audience           TEXT        NOT NULL DEFAULT 'Game developers and QA engineers',
    problem_and_escalation      TEXT,       -- any issues encountered during the test run

    -- Section 2: Summary
    test_items_summary          TEXT,       -- narrative from LLM about what was tested
    test_environment_summary    TEXT,       -- ViZDoom version, OS, hardware, model used
    hardware_spec               JSONB,      -- {"cpu": "...", "ram_gb": 16, "os": "Ubuntu 24"}
    software_spec               JSONB,      -- {"vizdoom": "1.2.x", "python": "3.11", "llm": "gemini-2.5-flash"}

    -- Section 3: Variances
    variances_from_plan         TEXT,       -- deviations: e.g. "Agent could not access MAP02 due to missing key"
    test_procedure_variances    TEXT,
    test_case_variances         TEXT,

    -- Section 4: Comprehensive Assessment
    test_coverage_evaluation    TEXT,       -- LLM narrative on coverage quality
    objectives_planned          JSONB,      -- list of planned objectives
    objectives_covered          JSONB,      -- which were achieved
    objectives_omitted          JSONB,      -- which were not reached
    uncovered_attributes        TEXT,       -- surprising/new behaviors found
    test_process_changes        TEXT,       -- recommendations

    -- Section 5: Summary of Results
    -- Defects stored in separate table (see below), this stores the narrative
    defect_summary_narrative    TEXT,
    defect_patterns             TEXT,       -- patterns across defect types

    -- Section 6: Evaluation
    test_item_limitations       TEXT,       -- incomplete features, limitations found
    dropped_features            TEXT,
    pass_fail_summary           JSONB,      -- {"map_navigation": "PASS", "combat_engagement": "FAIL", ...}
    risk_areas                  JSONB,      -- [{"area": "low ceiling corridors", "risk": "medium", "reason": "..."}]
    good_quality_areas          JSONB,      -- [{"area": "open arenas", "assessment": "agent performs well"}]

    -- Section 7: Summary of Activities
    major_activities_summary    TEXT,
    activity_variances          TEXT,
    elapsed_time_seconds        INTEGER,
    total_actions_taken         INTEGER,    -- copied from test_runs for report convenience

    -- PDF
    pdf_path                    TEXT,       -- absolute path on disk
    generated_at                TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    -- 'generating' | 'complete' | 'failed'
    generation_status           VARCHAR(16) NOT NULL DEFAULT 'generating',
    generation_error            TEXT
);
```

### Table 8: `defects`

Each defect found during testing — mapped from game events. This feeds Section 5 of the report.

```sql
CREATE TABLE defects (
    id                  UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    run_id              UUID        NOT NULL REFERENCES test_runs(id) ON DELETE CASCADE,
    report_id           UUID        REFERENCES test_reports(id) ON DELETE SET NULL,

    -- Defect classification matching QA template
    -- Severity: 1=Critical, 2=Major, 3=Minor, 4=Trivial
    severity            SMALLINT    NOT NULL CHECK (severity BETWEEN 1 AND 4),
    -- Priority: 1=High, 2=Medium, 3=Low
    priority            SMALLINT    NOT NULL CHECK (priority BETWEEN 1 AND 3),

    -- 'open' | 'resolved' | 'deferred'
    resolution_status   VARCHAR(16) NOT NULL DEFAULT 'open',

    defect_type         VARCHAR(64) NOT NULL,
    -- e.g. 'repeated_death_location' | 'softlock_navigation' | 'ammo_starvation'
    --      'unreachable_area' | 'health_deficit' | 'stuck_geometry'

    title               VARCHAR(255) NOT NULL,
    description         TEXT        NOT NULL,
    -- What the agent was doing, observed from game_events + LLM reasoning
    reproduction_steps  TEXT,
    -- The tick number where this defect was detected
    detected_at_tick    INTEGER,
    -- Position in map where defect occurred
    position_x          REAL,
    position_y          REAL,
    -- Link to the screenshot if one was captured at this moment
    screenshot_id       UUID        REFERENCES notable_event_screenshots(id) ON DELETE SET NULL,

    -- LLM's proposed fix/recommendation
    recommendation      TEXT,

    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_defects_run_id ON defects(run_id);
CREATE INDEX idx_defects_severity ON defects(run_id, severity);
```

**Defect mapping logic (your service layer, not DB):**
- Player death at same (x,y) ±50 units more than once → `repeated_death_location`, Severity 2
- Agent stuck (position unchanged for >30 ticks) → `stuck_geometry`, Severity 1
- Ammo hits 0 for >60 ticks → `ammo_starvation`, Severity 2
- HP drops below 10% for >30 ticks without recovery → `health_deficit`, Severity 3
- Map not completed + no progress for final 120 ticks → `softlock_navigation`, Severity 1
- Secret sectors exist in static analysis but `secret_count` never incremented → `unreachable_area`, Severity 3

## Full Entity Relationship Summary

```
wad_files
    └── static_analysis_results (1 wad → many maps analyzed)
    └── test_runs (1 wad → many runs)
            └── game_events (1 run → thousands of events)
            │       └── notable_event_screenshots (1 event → 0 or 1 screenshot)
            └── agent_position_trail (1 run → ~500 sampled positions)
            └── test_reports (1 run → exactly 1 report)
                    └── defects (1 report → many defects)
```

## What Goes in Local Storage (Browser)

NOT in PostgreSQL. localStorage holds only:

```json
{
  "recent_run_ids": ["uuid1", "uuid2", "uuid3"],
  "recent_wad_ids": ["uuid1"],
  "ui_preferences": {
    "default_difficulty": 3,
    "show_llm_reasoning": true
  }
}
```

The frontend uses these UUIDs to call your API and get fresh data. It does NOT cache actual run data in localStorage — that would go stale and be unreliable. localStorage is only a bookmarks list.
